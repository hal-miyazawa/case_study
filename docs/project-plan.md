# Project Plan

このファイルは、防災体験アプリの企画メモです。今後、実装者・企画者が読み返して、アプリの目的、体験設計、最小完成版の範囲を理解できるように残しています。

## 企画概要

本アプリは、災害発生前の限られた時間で部屋と持ち物を備え、災害発生後にその備えの結果を体験する、シナリオ分岐型の防災体験アプリです。

ユーザーが行った事前準備は内部フラグとして保持し、災害発生後の部屋の状態、移動可否、シナリオ、エンディング分岐に反映します。

目指す体験は、「備えたかどうか」が単なるチェック項目ではなく、ゲーム内の結果として返ってくることです。

## コアコンセプト

- 2D 主観視点で部屋を探索する
- 家具や持ち物をクリックして、防災対策を行う
- 災害前の行動には回数・時間制限を設ける
- 備えた内容によって、災害後の部屋やシナリオが変わる
- シナリオゲームのように分岐し、最終的に True End を目指す
- 教材らしさよりも、物語と体験の中で防災意識を促す

## 目的

防災アプリは、チェックリスト形式だけだと「面倒」「自分ごとにしづらい」と感じられやすいです。この企画では、シナリオと選択の結果を使って、防災への備えを自然に体験できる形にします。

ユーザーに伝えたいこと:

- 災害時に実際に起こりうる状況をイメージしてもらう
- 事前の備えが災害後の行動にどう影響するか体験してもらう
- 家具固定、防災バッグ、避難経路確認などの重要性を理解してもらう
- 現実の自分の部屋や生活にも防災対策を促す

特に、普段は想像しにくい災害時の状況を、シナリオとしてリアルに体験させることを重視します。

## 主人公設定

主人公は、仮設定として「予知能力者」とします。

ただし、未来を完全に見られるわけではなく、断片的なビジョンだけを見ることができます。

ビジョン例:

- 倒れた家具
- 塞がれた玄関
- 割れたガラス
- 暗くなった部屋
- 連絡が取れないスマホ
- 混雑した避難所

ユーザーは、この断片的な情報をもとに、災害前の準備内容を選びます。

## ゲームの流れ

### 1. 災害前の準備パート

災害が起こる前の限られた時間で、部屋や持ち物の防災対策を行います。

準備アクション例:

- 家具を固定する
- 玄関前に倒れそうな家具を置かない
- 靴やスリッパを用意する
- 懐中電灯を用意する
- 水や非常食を準備する
- モバイルバッテリーを用意する
- 防災バッグを準備する
- 避難経路を確認する

すべての対策を一度にできないように、行動回数や日数の制限を設けます。ユーザーは、見えたビジョンや部屋の状態をもとに、優先順位を考えて備える必要があります。

### 2. 災害発生パート

災害が発生すると、事前に行った備えに応じて部屋の状態が変化します。

分岐例:

- 家具固定をしていない: 家具が倒れて玄関が塞がる
- 靴を用意していない: ガラスが散乱して進めない
- ライトを用意していない: 停電で部屋が暗く、探索しづらい
- 防災バッグを用意していない: 外に出た後の対応が難しくなる

内部処理では、ユーザーの選択をフラグとして保持し、そのフラグに応じて表示オブジェクトやシナリオを切り替えます。

### 3. 災害後の避難パート

第二段階の拡張として、外に出た後に、防災バッグの中身で状況に対処しながら進む展開を想定します。

分岐例:

- 水を持っている: 体力を維持できる
- 懐中電灯を持っている: 暗い道を進める
- モバイルバッテリーを持っている: 情報収集や連絡ができる
- 非常食を持っている: 避難生活で困りにくい
- 必要なものが足りない: 避難途中や避難所生活で困る

部屋の中だけでなく、外に出た後にも事前準備の結果を反映させます。

## シナリオ・演出方針

### 教材っぽくしすぎない

「家具を固定しましょう」「水を用意しましょう」と説明するだけではなく、ゲーム内の出来事として重要性を伝えます。

例:

- 靴がないため、割れたガラスの上を進めない
- ライトがないため、停電後の部屋を安全に探索できない
- 家具固定をしていないため、玄関前が塞がる

### リアルな危機感を持たせる

実際に起こりうるが、普段は想像しにくいシチュエーションを入れます。

ただし、死や危険な状況を軽く扱うのは避けます。ゲームオーバーにする場合も、次の点を伝えることを重視します。

- どの備えが足りなかったのか
- なぜその状況になったのか
- 現実ではどう対策すればよいのか

### 家族や大切な人を絡める

将来的には、主人公や家庭状況を複数用意することを検討します。

シナリオ例:

- 一人暮らしの学生
- 家族と住んでいる人
- 高齢の家族がいる人
- 小さな子どもがいる家庭
- ペットがいる家庭

状況が変わることで、災害時に考えるべき備えや行動も変わります。

### 第三の壁への干渉

ゲームの最後に、ユーザー自身へ問いかける演出を入れます。

例:

> これは物語の中の部屋でした。
> では、実際のあなたの部屋はどうですか？

ゲーム内の体験から、現実の自分の部屋や生活へ意識を向けさせることが重要です。

## エンディング分岐案

### Bad End

部屋から出られない。

例:

- 家具が倒れて玄関が塞がる
- ガラスが散乱して移動できない
- 暗闇で安全に行動できない

### Normal End

脱出はできるが、避難生活で困る。

例:

- 水や食料がない
- 連絡手段がない
- 避難所で必要なものが足りない

### Good End

避難所まで到達できる。最低限の備えができており、自分の安全は確保できる。

### True End

自分だけでなく、家族や近所の人も助けられる。自助だけでなく、共助までつながるエンディングにする。

### やり直し要素

エンディング後に「やり直す力があれば……」のような演出を入れ、もう一度災害前に戻って対策を見直せるようにします。

予知能力者という設定と相性がよく、自然に再プレイへ誘導できます。

## 演出案

- 地震発生時に画面を少し揺らす
- 暗転する
- 効果音を鳴らす
- 家具が倒れた画像に差し替える
- テキストを一文字ずつ表示する
- 選択肢を少し遅れて出す
- 背景色を暗くする

処理自体はシンプルでも、演出によって体験のリアルさや緊張感を高めます。

## 最小完成版

まずは、次の流れが成立するところまでを最小完成版とします。

1. 部屋画面を表示する
2. 家具・備えを選択できる
3. 選択内容をフラグとして保存する
4. 地震発生シーンを表示する
5. フラグに応じて家具表示を変更する
6. シナリオを分岐させる
7. 結果画面を表示する

この段階で、「備えた内容が災害後の結果として返ってくる」という基本体験を成立させます。

最小版の具体的な画面遷移と内容は [story-flow.md](story-flow.md) にまとめています。

## AI 活用方針

背景や一部の素材制作には AI を活用する想定です。

AI を使うことで背景制作のコストを下げ、より多くのシーン差分を用意できます。その結果、災害前後の変化を細かく表現しやすくなります。

ただし、ゲームとして破綻しないように、固定する部分と差分化する部分を分けます。

固定するもの:

- 部屋の間取り
- 玄関の位置
- 本棚の位置
- ベッドの位置
- キッチンや窓の位置
- 視点

差分化するもの:

- 家具が倒れているか
- ガラスが割れているか
- 停電しているか
- 通路が塞がっているか
- 外の状況
- 避難所の混雑感

## 実装メモ

React / TypeScript / Vite で実装する想定です。

内部的には、ユーザーの選択を次のようなフラグとして持ち、災害後の表示やシナリオ分岐に使います。

```ts
type PreparedFlags = {
  fixedFurniture: boolean
  clearedEntrance: boolean
  preparedShoes: boolean
  preparedLight: boolean
  preparedWater: boolean
  preparedFood: boolean
  preparedBattery: boolean
  preparedEmergencyBag: boolean
  checkedEvacuationRoute: boolean
}
```

エンディング判定も、まずはこのフラグの組み合わせでシンプルに実装します。

## まとめ

本企画は、ユーザーが災害前に行った備えを内部フラグとして保持し、災害発生後の部屋の状態やシナリオ分岐に反映する、防災体験型アプリです。

チェックリストのように防災知識を一方的に伝えるのではなく、ユーザー自身の選択によって災害後の状況が変化することで、備えの重要性を体験として理解できる点が特徴です。

最終的には、ゲーム内の体験を通して、ユーザーが現実の自分の部屋や生活の防災対策を見直すきっかけを作ることを目指します。
